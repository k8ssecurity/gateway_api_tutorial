# Testing the Lab — Quick Guide

After `./setup.sh` finishes, you have two distinct traffic paths to exercise:

1. **Envoy Gateway** routing the sample webapp (HTTP + HTTPS, hostname `webapp.local.dev`).
2. **Agentgateway** routing the Microsoft Learn MCP server (`/mcp-mslearn`), driven by the OpenAI Agents SDK.

This guide walks both, with separate macOS and Linux paths where it matters.

---

## 0. Sanity check (any platform)

Run this first. If the output doesn't match, the rest of the guide won't work.

```bash
# Two GatewayClasses, both Accepted
kubectl get gatewayclass

# Two Gateways, both Programmed, with MetalLB IPs assigned
kubectl get gateway -A

# HTTPRoutes attached to their gateways
kubectl get httproute -A
```

Expected:

```
NAME           CONTROLLER                                       ACCEPTED   AGE
agentgateway   agentgateway.dev/agentgateway                    True       ...
eg             gateway.envoyproxy.io/gatewayclass-controller    True       ...

NAMESPACE              NAME                  CLASS          ADDRESS          PROGRAMMED   AGE
agentgateway-system    agentgateway-proxy    agentgateway   172.x.255.201    True         ...
envoy-gateway-system   eg-gateway            eg             172.x.255.200    True         ...

NAMESPACE             NAME           HOSTNAMES              AGE
agentgateway-system   mcp-mslearn                           ...
demo-app              webapp-route   ["webapp.local.dev"]   ...
```

If any of the above is missing or `PROGRAMMED=False`, jump to [Troubleshooting](#troubleshooting) before going further.

```bash
# Capture the Envoy Gateway IP for later commands
export GATEWAY_IP=$(kubectl get gateway/eg-gateway -n envoy-gateway-system -o jsonpath='{.status.addresses[0].value}')
echo "Envoy Gateway IP: $GATEWAY_IP"
```

## 1. Test the Envoy Gateway sample webapp

The sample app is `hashicorp/http-echo` returning `Hello from Gateway API!`. Three stable replicas, one canary replica, exposed via `HTTPRoute webapp-route` on hostname `webapp.local.dev`.

### Linux

The MetalLB IP is reachable directly from your shell:

```bash
# HTTP via hostname (setup.sh wrote /etc/hosts → $GATEWAY_IP)
curl http://webapp.local.dev/

# HTTP via IP + Host header
curl -H 'Host: webapp.local.dev' http://$GATEWAY_IP/

# HTTPS via hostname (-k skips self-signed cert validation)
curl -k https://webapp.local.dev/

# HTTPS via IP
curl -k -H 'Host: webapp.local.dev' https://$GATEWAY_IP/
```

All four should print:

```
Hello from Gateway API!
```

### macOS

The MetalLB IP `172.x.x.x` is **not** reachable from your Mac terminal — Docker Desktop runs containers inside a Linux VM. Two working options:

**Option A — `kubectl port-forward`**

In one terminal:

```bash
SVC=$(kubectl -n envoy-gateway-system get svc \
  -l gateway.envoyproxy.io/owning-gateway-name=eg-gateway \
  -o jsonpath='{.items[0].metadata.name}')
kubectl -n envoy-gateway-system port-forward svc/$SVC 8080:80 8443:443
```

In a second terminal (setup.sh wrote `127.0.0.1 webapp.local.dev` to your `/etc/hosts`):

```bash
curl http://webapp.local.dev:8080/
curl -k https://webapp.local.dev:8443/

# Or send the Host header explicitly:
curl -H 'Host: webapp.local.dev' http://localhost:8080/
curl -k -H 'Host: webapp.local.dev' https://localhost:8443/
```

**Option B — `docker exec` into a kind node**

No port-forward needed; the kind nodes are on the same Docker network as the MetalLB IPs:

```bash
docker exec gateway-api-lab-control-plane \
  curl -s -H 'Host: webapp.local.dev' http://$GATEWAY_IP/

docker exec gateway-api-lab-control-plane \
  curl -sk -H 'Host: webapp.local.dev' https://$GATEWAY_IP/
```

### Inspect the TLS cert

**Linux:**

```bash
echo | openssl s_client -connect $GATEWAY_IP:443 -servername webapp.local.dev 2>/dev/null \
  | openssl x509 -text -noout | head -20
```

**macOS** (with port-forward running):

```bash
echo | openssl s_client -connect localhost:8443 -servername webapp.local.dev 2>/dev/null \
  | openssl x509 -text -noout | head -20
```

You'll see the self-signed cert generated by `setup.sh` (`CN=*.local.dev`, valid 365 days).

## 2. Test the Agentgateway → Microsoft Learn MCP path

This drives the public Microsoft Learn MCP server (`https://learn.microsoft.com/api/mcp`) through the agentgateway proxy at `/mcp-mslearn`, using the OpenAI Agents SDK as the test harness.

### 2.1 Port-forward the agentgateway proxy (both platforms)

```bash
kubectl -n agentgateway-system port-forward \
  deployment/agentgateway-proxy 8080:80
```

Leave that running. On Linux you could alternatively curl the MetalLB IP `172.x.255.201` directly, but port-forward keeps the rest of this section identical between platforms.

### 2.2 Set up a venv and install the OpenAI Agents SDK

```bash
cd gateway-api-lab
python3 -m venv .venv
source .venv/bin/activate
pip install --upgrade openai-agents
```

A fresh venv per lab session avoids dependency conflicts with other Python tooling on your machine.

### 2.3 Export your OpenAI API key and run the test

```bash
export OPENAI_API_KEY=sk-...
python3 test-mslearn-agent.py
```

What you should see:

1. On stderr, the list of tools Microsoft Learn MCP exposes (typically `microsoft_docs_search`, `microsoft_docs_fetch`, plus a few related tools).
2. On stdout, a cited multi-paragraph answer about Azure App Service with `learn.microsoft.com/...` URLs inline.

### 2.4 About the trailing "Session termination failed: 202"

After the agent answer, you'll see:

```
Session termination failed: 202
```

This is **cosmetic**. When the SDK closes the MCP session, Microsoft Learn responds with HTTP 202 (Accepted) rather than 200 (OK). The openai-agents SDK strictly expects 200 and logs the 202 as a "failure" — but the agent's answer has already been returned successfully before that line prints. You can ignore it.

### 2.5 (Optional) Restrict the tool list with an AgentgatewayPolicy

To prove gateway-level tool gating works, apply the commented-out policy at the bottom of `11-mcp-mslearn.yaml`, or apply this directly:

```bash
kubectl apply -f- <<'EOF'
apiVersion: agentgateway.dev/v1alpha1
kind: AgentgatewayPolicy
metadata:
  name: mslearn-tools-allowlist
  namespace: agentgateway-system
spec:
  targetRefs:
    - group: agentgateway.dev
      kind: AgentgatewayBackend
      name: mslearn-mcp-backend
  backend:
    mcp:
      authorization:
        action: Allow
        policy:
          matchExpressions:
            - 'mcp.tool.name == "microsoft_docs_search"'
EOF
```

Re-run `python3 test-mslearn-agent.py`. The tool list on stderr should now show only `microsoft_docs_search`, and the agent's answer will be noticeably shorter because it can no longer fetch full articles.

Remove the policy to restore full tool access:

```bash
kubectl delete agentgatewaypolicy mslearn-tools-allowlist -n agentgateway-system
```

## 3. Try the advanced Envoy Gateway routes

These are optional, but show off what HTTPRoute can do beyond the basic case. All commands below assume Linux-style `curl http://$GATEWAY_IP/` — on macOS, wrap with `docker exec gateway-api-lab-control-plane` (with `-H 'Host: webapp.local.dev'`) or curl `localhost:8080` while port-forwarded.

### 3.1 Traffic splitting (90% stable / 10% canary)

```bash
kubectl apply -f gateway-api-lab/07-httproute-canary.yaml

for i in {1..20}; do
  curl -s -H 'Host: webapp.local.dev' http://$GATEWAY_IP/
done
```

Roughly 2 out of 20 responses should say `CANARY VERSION - New Features!`.

### 3.2 Header-based routing

```bash
kubectl apply -f gateway-api-lab/08-httproute-header.yaml

# Default — stable
curl -H 'Host: webapp.local.dev' http://$GATEWAY_IP/

# X-Canary: true — canary
curl -H 'Host: webapp.local.dev' -H 'X-Canary: true' http://$GATEWAY_IP/
```

### 3.3 Reset to basic routing

```bash
kubectl apply -f gateway-api-lab/06-httproute-basic.yaml
```

### 3.4 TLS passthrough (TLSRoute)

See [Part 7.4 of the main tutorial](kubernetes-gateway-api-tutorial.md#74-tls-passthrough-with-tlsroute) — it deploys an extra TLS-enabled nginx backend with its own cert, plus a passthrough Gateway on port 8443. Useful to compare end-to-end-encrypted vs gateway-terminated TLS.

## Troubleshooting

### Gateway has `PROGRAMMED=False` or no `ADDRESS`

MetalLB hasn't assigned an IP. Check the pool:

```bash
kubectl get ipaddresspools -n metallb-system -o yaml
docker network inspect kind \
  -f '{{range .IPAM.Config}}{{.Subnet}}{{"\n"}}{{end}}' \
  | grep -E '^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+/[0-9]+$'
```

The pool's IP range must overlap the IPv4 subnet shown by `docker network inspect`. If the Docker network is on `172.20.x.x` but the pool is on `172.18.x.x`, MetalLB has nothing to advertise. Re-run `setup.sh` — it auto-detects the IPv4 subnet correctly.

### `kubectl port-forward` says "address already in use"

Another process is on the port. Pick a different host port:

```bash
kubectl -n envoy-gateway-system port-forward svc/$SVC 9090:80 9443:443
# then use http://localhost:9090/ and https://localhost:9443/
```

### Agent hangs on `list_tools()`

Either the port-forward isn't running, or the backend's `protocol:` doesn't match the upstream's transport.

```bash
# Is the port-forward listening?
lsof -iTCP:8080 -sTCP:LISTEN

# Are there agentgateway errors during the request?
kubectl logs -n agentgateway-system deployment/agentgateway --tail=100 -f
```

`11-mcp-mslearn.yaml` declares `protocol: StreamableHTTP`, which matches Microsoft Learn's transport. Don't change it to `SSE` unless you swap the upstream too.

### `OPENAI_API_KEY` is set but the agent fails authentication

The key may have expired or hit a rate limit. The test script prints the OpenAI SDK error directly — read the message and check at https://platform.openai.com/usage.

### "Hello from Gateway API!" never appears

Check the HTTPRoute is attached:

```bash
kubectl describe httproute webapp-route -n demo-app
```

Look for `Conditions: Accepted=True, ResolvedRefs=True` on the parent. If either is False, the message tells you why.

## Cleanup

```bash
cd gateway-api-lab
./cleanup.sh
# or:
kind delete cluster --name gateway-api-lab
```

That removes the entire cluster including the agentgateway and Envoy Gateway control planes. The `/etc/hosts` entry written by `setup.sh` remains — remove it by editing `/etc/hosts` and dropping the `webapp.local.dev api.local.dev` line.
